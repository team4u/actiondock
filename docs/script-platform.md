# ActionDock：脚本即 Skill

> 不再为每个脚本手写 Skill。让脚本本身成为可发布、可调用、可审计的工具能力。

---

## 自动化一开始都是脚本，后来都想变成工具

很多自动化最开始都是一个脚本。

它可能用来清理临时文件、同步数据、生成报表、检查服务状态，或者调用某个内部系统。刚开始，只要能跑通就够了。

真正的问题出现在它开始变得有用之后。

脚本被复制到服务器、CI、项目目录和文档里；参数改了，调用方没同步；依赖升级了，旧副本还在跑。出问题时，很难知道是谁在什么时候、用什么输入执行过。

后来，它可能还会被定时任务、Webhook、其他脚本，甚至 AI Agent 调用。

这个时候，脚本就不再只是一个临时命令，而是一块隐形的基础设施。

问题也随之变化：

> 脚本不只是要能跑，还要能被稳定、安全、可审计地调用。

ActionDock 要解决的，就是这个问题。

---

## “脚本即 Skill”到底是什么意思

在 ActionDock 里，“脚本即 Skill”不是说每个脚本都要生成一份独立的 Skill 文件。

它的意思是：

> 脚本自己的 Schema、发布版本、权限边界和执行记录，天然构成了 AI 可理解、可调用、可审计的工具契约。

一个脚本不再只是源码文件。它包含：

* 脚本 ID：工具标识
* `inputSchema`：输入参数契约
* `outputSchema`：返回结果契约
* 发布快照：稳定版本边界
* 执行记录：审计来源
* 权限和 Toolset：AI 可见范围

所以，ActionDock 不需要为每个脚本额外维护一套 Prompt 或 Skill 描述。

脚本定义本身，就是工具能力的来源。

---

## 脚本真正的问题：不是能不能跑，而是能不能长期可靠地被调用

一个脚本在本地运行时，问题通常很简单：输入是什么，输出是什么，能不能跑通。

但当它进入团队环境、生产环境或 AI Agent 的工具链之后，问题会变得复杂：

* 谁可以执行它？
* 它需要哪些参数？
* 哪些参数是必填？
* 返回值是什么结构？
* 调用失败时怎么排查？
* 它当前跑的是哪个版本？
* AI Agent 是否有权限调用它？
* 调用记录是否能回看？

传统做法通常是在脚本之外再维护一层说明：CLI 文档写一套，REST API 写一套，管理台表单写一套，AI Prompt 或 Skill 里再写一套。

问题是，脚本一改，这些说明很容易不同步。

ActionDock 的做法是把这些约定收回到脚本定义里。

脚本不是旁边再挂一个工具描述。脚本定义本身就是工具契约。

---

## ActionDock 的基础：一个脚本，一份 Schema，所有入口共用

ActionDock 把脚本的输入、输出和执行约定都收进脚本定义里。

一个示例脚本定义如下：

```json
{
  "id": "hello-groovy",
  "name": "Hello Groovy",
  "type": "GROOVY",
  "source": "def name = input.name ?: \"World\"\nreturn [message: \"Hello, \" + name + \"!\", upperName: name.toUpperCase()]",
  "inputSchema": {
    "type": "object",
    "properties": {
      "name": {
        "type": "string",
        "title": "Name"
      }
    }
  },
  "outputSchema": {
    "type": "object",
    "properties": {
      "message": {
        "type": "string",
        "title": "Message"
      },
      "upperName": {
        "type": "string",
        "title": "Upper Name"
      }
    }
  }
}
```

这份定义不是给某一个入口看的。

CLI 用它解析命令参数。
管理台用它生成表单。
REST API 用它校验输入输出。
AI Agent 用它理解工具应该怎么调用。

同一个脚本，可以通过 CLI 执行：

```bash
actiondock script run hello-groovy --name alice --json
```

也可以通过 REST API 调用：

```bash
curl -X POST http://localhost:5177/api/scripts/hello-groovy/execute \
  -H 'Content-Type: application/json' \
  -d '{"input":{"name":"alice"},"mode":"SYNC"}'
```

还可以在管理台里通过自动生成的表单执行。

Schema 变成人、系统和 AI 共同理解脚本的语言。

## 从一个真实场景看 ActionDock 的价值

假设有一个底层脚本叫 `query-log`。

它负责查询内部日志系统，接收服务名和时间范围，返回原始日志或聚合结果。

返回值结构大概是：

```json
{
  “logs”: [...],
  “total”: 1234,
  “queryTime”: “2026-05-13T10:20:00Z”
}
```

这个脚本本身很简单，但用得上。

有人会在它基础上写服务异常分析脚本，有人在 CI 里用它做部署验证，有人拿它接监控告警，有人让它成为 AI Agent 的调试工具。每个调用方都会在 `query-log` 的基础上加一层自己的逻辑。

问题来了：有一天日志系统升级了，API 变了。

日志格式改了，返回字段名变了，认证方式也变了。

如果是传统方式，每个调用方都得改自己本地的 `query-log` 副本。但问题是，没人知道这个脚本被复制到多少地方、跑了多少个版本、改了多少个字段。

最后只能靠”去服务器上搜搜看有没有这个脚本”，或者”祈祷没人用老版本”。

在 ActionDock 里，`query-log` 是一份发布在平台上的脚本。

当底层 API 变了，你只需要改这一份脚本，更新它的实现，重新发布。`query-log` 的 inputSchema 不变，调用方不需要任何改动——它们甚至不知道底层换了一套接口。

调用方拿到的是新版的日志结果，字段名对齐，认证通过，不需要任何额外的适配工作。

这就是”脚本即 Skill”的价值：把基础能力统一收进平台，让依赖方专注自己的业务逻辑，不用操心底层实现怎么变。

---

## 发布脚本，就是发布稳定工具能力

脚本越有用，越不能随手改。

一旦它被定时任务、Webhook、AI Agent 或其他脚本调用，随手改源码就会变成风险。可能只是想调试一个字段，但线上流程已经开始跑这份改动。

ActionDock 把脚本分成草稿和发布两个状态。

草稿用于开发、调试和试跑。发布后，平台会生成稳定快照。

定时任务、脚本互调、Webhook 和 AI 工具调用，应该尽量使用发布版本，避免草稿改动影响稳定流程。

这样脚本可以像小型工具一样演进：

* 开发时可以快速调整
* 发布后形成稳定边界
* 出问题时可以回看版本
* 需要时可以回退到上一个发布状态

发布脚本，本质上就是发布一项稳定的工具能力。

---

## 执行记录、日志和审计：脚本进入长期运行环境

自动化越有用，越需要知道它到底做过什么。

脚本被人手动执行、被定时任务触发、被 Webhook 调用、被 AI Agent 使用以后，如果没有统一记录，排查问题只能靠猜。

ActionDock 会保存脚本执行的关键信息：

* 谁触发了执行
* 使用了哪个脚本版本
* 传入了什么参数
* 返回了什么输出
* 执行状态是什么
* 日志里记录了哪些步骤
* 失败时错误详情是什么

外部 Webhook 调用脚本后，也会保存执行记录，脚本输入中会保留请求来源。

这让脚本不只是能跑，还能回看、能解释、能追责。

AI 调用也一样。

当 Agent 调用 ActionDock 中的脚本工具时，这次调用同样会进入平台的执行记录。你可以看到 AI 调用了哪个脚本、传了什么参数、脚本返回了什么结果，以及失败发生在哪一步。

## 配置值：让脚本适配不同环境

脚本不应该把环境信息硬编码在源码里。

不同环境可能有不同的服务地址、Token、数据库连接、监控系统地址和通知渠道。如果这些信息散落在脚本里，复制、更新和审计都会变得困难。

ActionDock 把环境信息集中到配置层。

脚本和插件可以通过 `config.get()` 或 `${config.key}` 引用配置。敏感值可以标记为 Secret，并在管理台和接口中脱敏。

这意味着你不需要为每台服务器维护一套脚本。脚本定义一致，Schema 一致，调用入口一致，区别只是配置值不同。

## 脚本互调和插件：从单个脚本到自动化能力网络

ActionDock 不只是管理单个脚本。

它还允许脚本调用其他脚本、调用插件、读取配置、写入日志和使用共享状态。

ActionDock 支持 Groovy 和 Python 两种脚本语言。

两种语言的脚本能力保持一致：平台注入的运行时对象、调用其他脚本、读写共享状态、读取配置、写入日志等能力，都可以按统一方式使用。

| 对象        | 作用        |
| --------- | --------- |
| `scripts` | 调用其他已发布脚本 |
| `plugins` | 调用平台插件动作  |
| `state`   | 读写共享状态    |
| `config`  | 读取全局配置值   |
| `log`     | 写入执行日志    |

Groovy 可以调用 Python 脚本，Python 也可以调用 Groovy 脚本。

Groovy 调用脚本：

```groovy
def result = scripts.invoke("python-data-脚本转换", [input: rawData])
```

Python 调用脚本：

```python
result = scripts.invoke("groovy-helper", {"input": rawData})
```

需要注意的是，`scripts.invoke(...)` 调用的是已发布脚本。目标脚本的入参和返回值需要与其已发布 Schema 对齐。

---

## 插件负责稳定能力，脚本负责编排流程

脚本适合表达流程，但不是所有能力都适合写在脚本里。

内部 SDK、专有系统接入、统一鉴权、复杂客户端封装、加密签名、消息队列连接、数据库客户端、文件格式解析、重试策略等能力，如果每个脚本都写一遍，很快会变成另一种复制。

ActionDock 提供插件机制。

插件适合沉淀那些不应该散落在脚本里的稳定能力。平台或基础设施团队可以把复杂系统接入做成插件，业务或运维同学再用脚本编排这些能力。

Groovy 调用插件：

```groovy
def result = plugins.invoke("my-plugin", "hello", [name: "world"])
```

Python 调用插件：

```python
result = plugins.invoke("my-plugin", "hello", {"name": "world"})
```

对脚本作者来说，底层能力来自 Groovy、Python 还是 Java 插件并不重要。

重要的是，平台里的能力都能按统一方式调用。

## 脚本也能反向调用 AI

除了让外部 AI 客户端通过 Skill 理解和调用脚本，ActionDock 自己也内置了 AI 模型和 Agent 能力。

这意味着脚本不再只能执行固定逻辑，也可以直接使用大模型。

过去，如果脚本里想用 AI，通常要自己接模型 API、管理 Key、拼请求、处理返回、记录用量和错误。每个脚本都接一遍，安全、审计和维护都会变散。

ActionDock 把模型配置收进平台。脚本可以通过内置 AI 能力调用聊天、结构化输出、向量或 Agent。

例如：

* 告警处理脚本可以先拉取监控数据，再调用大模型总结异常原因
* 工单脚本可以读取用户描述，让模型提取字段并判断优先级
* 报表脚本可以生成结构化数据后，再让模型写出面向业务人员的摘要

这让脚本从“固定命令”进一步变成“可编排 AI 能力的自动化单元”。

---

## 一个内置 Skill，串起整个脚本生命周期

当脚本有了 Schema、发布版本、执行记录、配置层、Profile、插件和 AI 能力之后，AI 调用脚本就不需要再靠临时 Prompt 猜测。

ActionDock 也不需要为每个脚本生成一份独立 Skill 文件。

平台只需要提供一个内置 ActionDock Skill。

这个内置 Skill 负责让 AI 客户端理解 ActionDock 的统一入口，并通过脚本定义动态发现和调用平台里的能力。

它串起的是整个脚本生命周期：

```text
发现脚本 → 读取 Schema → 生成参数 → 执行脚本 → 查看结果和日志 → 必要时生成新脚本草稿
```

例如：

```text
用户：帮我查一下有哪些已发布脚本
  ↓
AI：通过 ActionDock Skill 调用 actiondock script list
  ↓
用户：执行 hello-groovy，参数 name=alice
  ↓
AI：读取脚本定义，生成参数，调用 actiondock script run
  ↓
ActionDock：返回执行结果和日志
```

内置 Skill 负责统一入口。

脚本定义负责描述工具能力。

发布版本负责稳定边界。

执行记录负责审计追踪。

这就是 ActionDock 里 Skill 的核心价值：不是让你多写一层提示词，而是把“生成脚本、发布脚本、调用脚本、追踪执行”收进同一套工程体系。

---

## 如果脚本还不存在，AI 可以按 ActionDock 规范生成草稿

内置 Skill 不只可以调用已有脚本。

如果脚本还不存在，它也可以辅助生成脚本草稿。

你可以直接说：

```text
帮我写一个脚本，输入服务名和环境，检查健康接口。
如果失败，返回失败原因；如果成功，返回状态、耗时和检查时间。
```

AI 可以按 ActionDock 的规范生成：

* 脚本源码
* `inputSchema`
* `outputSchema`
* 示例入参
* 使用说明
* 可能的错误处理逻辑

生成结果不是直接上线。

它会先成为草稿。你可以在管理台里校验、试跑、调整，然后再发布成稳定版本。

发布之后，它又可以被内置 Skill 发现和调用。

这让 ActionDock 不只是一个“让 AI 调用脚本”的平台，也是一套“让 AI 协助生成、调试、发布和复用脚本”的体系。

## AI 暴露方式一：动态模式

ActionDock 有两种向 AI 暴露脚本的方式。

第一种是动态模式。

在动态模式下，AI 通过内置 ActionDock Skill 查询脚本列表、读取脚本定义，再调用统一的 runScript 能力执行目标脚本。

它的优势是：

* 不需要为每个脚本维护一份 Skill
* 脚本更新后，AI 可以读取最新发布定义
* 适合内部自动化脚本较多的平台型场景

动态模式的核心是：

> AI 先理解 ActionDock，再通过 ActionDock 理解具体脚本。

---

## AI 暴露方式二：工具化模式

第二种是工具化模式。

在工具化模式下，平台把被授权的 TOOL 类型脚本直接暴露为 Agent 工具。

工具名可以是：

```text
script.<scriptId>
```

脚本的 `inputSchema` 会直接变成工具输入结构，Agent 不需要靠 Prompt 猜参数。

这种方式适合稳定、高频、明确要加入 Agent Toolset 的脚本。

例如：

* `script.check-service-health`
* `script.notify-team`
* `script.create-incident-ticket`
* `script.generate-weekly-report`

工具化模式的核心是：

> 常用脚本可以直接成为 Agent Toolset 里的标准工具。

动态模式和工具化模式并不冲突。

它们共用同一份脚本定义、Schema、发布快照和执行记录，只是暴露方式不同。

---

## AI 能调用什么，必须有边界

脚本即 Skill，不等于所有脚本都能被 AI 随便执行。

在 ActionDock 中，脚本需要先发布，再显式标记为 TOOL 类型，并加入指定 Toolset，才会暴露给 Agent。

不同 Agent Profile 可以绑定不同 Toolset。

这意味着不同模型、不同场景、不同用户看到的工具范围可以不同。

AI 可以调用脚本，但只能在平台定义好的工具边界、权限边界和审计边界内调用。

这个边界非常重要。

它避免了“AI 可以执行脚本”变成“AI 可以任意执行所有脚本”。

---

## Agent Profile：模型、工具和任务知识的组合

在 ActionDock 中，Agent Profile 由几类内容组成：

* Model Profile：使用哪个模型、哪个供应商、什么 API Key
* Toolset / Direct Tools：Agent 能调用哪些工具
* Skill ：Agent 具备哪些任务知识、流程约束和使用说明

例如：

```text
# 告警处理

## 背景
收到服务告警后，需要判断告警级别，并按不同流程处理。

## 可用工具
- check-service-health：检查服务健康状态
- query-service-metrics：查询服务监控指标
- notify-team：发送团队通知

## 流程
1. 使用 check-service-health 检查服务状态
2. 使用 query-service-metrics 查询最近 15 分钟指标
3. 根据错误率、延迟和实例状态判断告警级别
4. 如果达到 P1 或 P2，使用 notify-team 通知对应团队
5. 返回处理摘要和建议动作
```

业务 Skill 描述的是“什么时候、按什么规则去做”。

ActionDock 脚本定义描述的是“这个工具怎么调用”。

两者分工明确，维护成本更低。

## 工具多了以后，发现和更新才是问题

工具变多后，麻烦的不是“有没有”，而是“别人怎么发现、怎么安装、怎么更新、怎么知道自己装的是不是同一版”。

ActionDock 的 Repository 机制用来解决这个问题。

仓库里可以放：

* 脚本工具
* 插件
* Webhook资产
* AI 能力包
* Skills
* 调度模板
* 配置模板

仓库可以来自 Git、HTTP 或本地目录。

同步后，用户可以从管理台或 CLI 发现、安装和更新这些资产。

仓库资产不只包含源码，也可以带上依赖、调度模板和配置模板。安装时可以选择是否连同依赖一起安装，避免“脚本装好了，环境还缺一堆东西”。

仓库也支持工作副本与上游同步。

仓库里的工具和Webhook资产可以创建为本地工作副本。平台会通过上游绑定记录来源、版本、路径、提交和摘要，并判断当前状态是：

* 已同步
* 本地有改动
* 远端有改动
* 两边同时有改动

这样团队可以在平台里直接编辑和调试本地工作副本，同时保留与仓库上游的同步关系和更新通道。

## 一个入口，所有服务器

ActionDock 的 CLI Profile 让同一个入口可以指向不同服务器或环境。

例如，你可以在一个聊天窗口里和 AI 协作：

```text
帮我读取下A电脑的文档1和B电脑的文档2，看看有哪些明显的差异
```

AI 通过 ActionDock Skill 使用 `a` Profile，先在a电脑执行脚本并返回结果。
然后，AI 只需要把目标 Profile 从 `a` 换成 `b`，就可以无缝地在b电脑上执行另一个脚本。

两次调用指向两台不同机器，但入口始终一致：

```text
actiondock script run <scriptId> --profile <profileName>
```

你不需要登录每个节点，只需要在一个入口即可完成所有操作。

---

## 与传统 Skill 的区别

传统 Skill 通常是在已有脚本之外再维护一层工具描述。

你要告诉模型有哪些脚本、每个脚本叫什么、参数怎么传、哪些字段必填、返回结果怎么看、什么时候可以调用。

这会带来重复维护问题。

脚本本身维护一套逻辑，Skill 或 Prompt 里再维护一套说明。一旦脚本参数变化，而 Skill 说明没有同步，AI 就可能继续按旧方式调用。

ActionDock 的思路不同。

它不是为每个脚本手写一份 Skill，而是让脚本定义直接成为工具契约。

| 场景      | 传统 Skill          | ActionDock             |
| ------- | ----------------- | ---------------------- |
| 每个工具的描述 | 需要手写和维护           | 由脚本定义和 Schema 提供       |
| 参数说明    | 在 Skill 里另写一套     | 使用脚本自己的 `inputSchema`  |
| 返回值说明   | 在 Skill 里另写一套     | 使用脚本自己的 `outputSchema` |
| 版本管理    | 脚本和 Skill 容易不同步   | 发布快照是统一稳定边界            |
| 多入口调用   | CLI、API、AI 各自维护说明 | 共用同一份脚本定义              |
| AI 调用流程 | 依赖 Prompt 描述工具    | 通过内置 Skill 动态发现和执行     |
| 审计追踪    | 通常需要额外实现          | 执行记录进入平台统一审计           |

ActionDock 的目标不是让脚本多一层 Prompt。

它要让脚本本身成为可发布、可复用、可被 AI 正确调用的工具能力。

## 与拖拽式工作流的区别

n8n、Dify、扣子这类拖拽式工具，适合把已有节点快速串起来。

它们上手快，适合业务人员配置流程，也适合验证想法。

ActionDock 面向的是另一类场景：那些最终会沉淀成脚本、内部工具或平台能力的自动化流程。

这类流程通常会慢慢变复杂：

* 条件分支越来越多
* 异常处理越来越细
* 依赖需要复用
* 配置需要区分环境
* 代码需要评审
* 版本需要回退
* 执行需要审计
* 工具需要被 AI Agent 调用

这时，用代码表达逻辑、用 Schema 描述契约、用发布快照管理稳定版本，会比把所有逻辑堆在画布上更适合长期维护。

| 场景          | 拖拽式工作流       | ActionDock     |
| ----------- | ------------ | -------------- |
| 主要使用者       | 非技术人员、业务配置人员 | 开发、运维、平台人员     |
| 基础表达方式      | 画布和节点        | 脚本、Schema、插件   |
| 复用单元        | 节点           | 脚本和插件          |
| 复杂逻辑        | 更适合轻量编排      | 更适合用代码表达       |
| 版本边界        | 依平台能力而定      | 发布快照作为稳定边界     |
| AI Agent 调用 | 通常需要整理工具描述   | 基于 Schema 暴露工具 |

ActionDock 不是要替代所有可视化编排工具。

它补的是另一层：把那些已经或即将变成脚本、服务、内部工具的流程，放进一套更贴近开发者工作方式的工程体系里。

## 快速开始

安装并启动：

```bash
npm install -g actiondock
actiondock server
```

启动后访问管理台：

```text
http://localhost:5177/admin/app/scripts
```

运行一个内置示例：

```bash
actiondock script run hello-groovy --name alice --json
```

成功后会看到类似结果：

```json
{
  "status": "SUCCESS",
  "output": {
    "message": "Hello, alice!",
    "upperName": "ALICE"
  }
}
```

接下来可以继续尝试：

* 创建一个脚本，补充输入输出 Schema
* 在管理台里试跑脚本草稿
* 发布脚本，生成稳定快照
* 用内置 Skill 让 AI 发现和调用已发布脚本
* 让 AI 按 ActionDock 规范生成脚本草稿
* 在脚本里调用其他脚本、插件或 AI 能力
* 为脚本添加定时任务或 Webhook
* 配置不同环境的 CLI Profile
* 把脚本发布到仓库，供其他人发现、安装和更新

## 总结

ActionDock 的核心很简单：脚本即 Skill。

但这里的 Skill 不是额外手写的一层 Prompt，也不是为每个脚本维护一份独立说明。

它来自脚本本身。

脚本有 `inputSchema`，所以 AI 知道参数怎么传。
脚本有 `outputSchema`，所以系统知道返回值怎么看。
脚本有发布快照，所以调用方有稳定版本边界。
脚本有权限和 Toolset，所以 AI 只能看到被授权的工具。
脚本有执行记录，所以每次调用都能回看、解释和审计。

ActionDock 做的不是把脚本包装成另一个概念。

它做的是让脚本从临时命令变成稳定工具：

> 有 Schema，有发布版本，有权限边界，有执行记录，也能被 AI 正确发现和调用。

脚本仍然是脚本。

但发布之后，它就是一个可复用、可审计、可被 Agent 使用的 Skill。

对个人来说，ActionDock 是统一的脚本管理和自动化入口。

对团队来说，它可以承接脚本分发、环境隔离、执行审计、插件复用、仓库管理和 AI 工具化。

脚本不缺。

真正缺的是让脚本直接成为 Skill，并且长期可靠运行的工程体系。

ActionDock 补上的，就是这一层。
